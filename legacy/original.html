API Rate Limiter · SMTP 자원 남용 방지

공개 API에 요청 제한이 없어, 공격자가 서로 다른 이메일로 대량 요청 시 Gmail SMTP 일일 한도(100건)가 2분 내 소진되어 전체 서비스가 중단될 위험이 있었습니다. 동일 이메일에 대한 3분 쿨다운만 존재하고 IP 기반 제한은 전혀 없었습니다.

AI가 만드는 영역을 자동화할수록, 사람이 해야 할 설계와 구조화의 영역이 더 선명해지는 경험이었습니다.

만드는 것보다 운영하면서 드러나는 문제를 해결하는 것이 엔지니어의 본질적인 역할이라고 생각합니다.

구독자 피드백을 바탕으로 콘텐츠 구조와 발송 주기를 지속적으로 개선하고 있으며, 구독·인증·발송이라는 결제/배송과 유사한 트랜잭션 플로우를 직접 설계하고 운영한 경험이 있습니다.

4일 만에 런칭한 것은 의도적인 트레이드오프였고, 이후 기술 부채를 체계적으로 해소하며 속도와 확장성의 균형을 맞춰가고 있습니다. 자동화할수록 새로운 과제가 열리고, 그 과제를 해결하면 또 다음 자동화가 가능해지는 순환을 만들어가고 있습니다.

문제

동시 70명만 접속해도 서비스가 사실상 멈추는 상태였습니다. Application 내부에서 모든 row에 대해 Haversine 삼각함수 거리 계산을 수행하는 구조가 Full Table Scan과 결합되어, 요청마다 전체 데이터를 수학 연산으로 필터링하고 있었습니다.

시도 1
"이 정도면 됐지?"라는 유혹

Spatial Index + ST_Distance_Sphere로 거리 계산을 DB에 위임했습니다. TPS 4에서 50으로 12배 개선. 여기서 만족할 뻔했지만, 동시 요청을 더 올려보니 DBCP가 고갈되기 시작했습니다. 테스트 환경에서는 문제없어 보였지만, 실제 부하에서는 버티지 못하는 수치였습니다.

시도 2
Redis GeoHash 도입

DB 의존도를 완전히 끊어야 한다고 판단했습니다. Redis GeoHash는 2차원 좌표를 Z-order curve 기반 1차원 문자열로 변환하여 Sorted Set에 저장하는 구조로, 메모리 레벨에서 O(log N) 범위 검색이 가능합니다.

읽기 비율이 압도적으로 높은 매칭 조회 특성상 캐시 레이어 도입이 적합하다고 판단했고, MySQL은 원본 저장소, Redis는 조회 전용 레이어로 책임을 분리했습니다. 데이터 정합성은 매칭 등록/수정 시 Redis에 동기 반영하는 방식으로 보장했습니다.

검증

AWS t3.small, MySQL 8.0, Redis 7.0 환경에서 반경 5km 매칭 검색 시나리오로 부하 테스트를 수행했습니다. TPS가 4에서 440으로 110배 개선되었고, 평균 응답 시간은 5000ms에서 25ms로 단축되었습니다.

데이터가 없으면 "됐다"고 말하지 않습니다.

처음에는 Spatial Index만으로 충분하다고 생각했지만, 부하를 더 올려보니 DBCP가 고갈되기 시작했고, 그제서야 테스트 결과에만 안심하고 있었다는 걸 알게 됐습니다.

이 경험으로 트래픽 증가에 대응하는 캐시 레이어 도입 판단 기준과 데이터 정합성 전략을 실전에서 배웠습니다. 현재 Redis가 단일 장애점(SPOF)이라는 한계가 남아있고, Sentinel/Cluster 구성으로 해결할 계획입니다.

문제

현장 인력이 하루 5~10회 직접 출동해 PC를 재부팅해야 하는 상황이었습니다.

병원 스피드게이트(QR 인식 차단문)의 QR 인식을 담당하는 소프트웨어는 병원 인트라넷 웹앱의 "QR 인증기" 기능이었는데, 간헐적으로 WiFi가 끊기면 인트라넷 전체가 로그아웃되면서 QR 인증기 화면까지 함께 사라졌습니다. 화면이 꺼진 상태에서는 QR을 아무리 갖다 대도 인증이 되지 않았습니다.

해결
복합 소프트웨어 의존을 끊고 단일 목적 앱으로 분리

먼저 병원 측에 WiFi 안정화를 요청했지만 "원인 파악이 어렵다"는 답변이 돌아왔습니다. 기존 인트라넷은 복합 소프트웨어라 QR 인증만을 위한 복구 로직을 넣기도 어려웠습니다.

2~3명의 소규모 팀에서 제가 직접 병원 서버팀과 소통하여 QR 인증 API를 별도로 요청받고, QR 인증만 수행하는 경량 앱을 직접 개발하여 스피드게이트 미니 PC에 설치했습니다.

WiFi가 끊겨도 재연결될 때까지 자동 재시도하도록 설계했고, 실제로 WiFi가 1분 이내로 복구되는 패턴이었기 때문에 이 방식으로 충분했습니다.

검증

로컬에서 강제 WiFi 차단 테스트 후 점진적으로 배포했습니다. 전체 적용 이후 주 단위 모니터링 결과를 팀에 공유하며, 하루 5~10회이던 현장 출동이 주 1~2회 수준으로 감소했음을 데이터로 확인시켰습니다.

완전히 0회가 되지 못하는 이유는, WiFi가 장시간 완전 단절되는 드문 경우에는 물리적 재부팅이 불가피한 현실적 한계가 있기 때문입니다.

통제할 수 없는 환경을 탓하기보다, 통제할 수 있는 범위에서 최선을 찾습니다. 병원 WiFi는 우리가 고칠 수 없었습니다. "근본 원인을 못 고치면, 근본 원인에 의존하지 않는 구조를 만들면 되지 않나?"로 발상을 바꾸니 길이 보였습니다. 완벽한 해결은 아니었지만, 출동 횟수가 눈에 띄게 줄었을 때 방향이 맞았다는 걸 확인할 수 있었습니다.

사내 프레임워크 코드 개선 · 중복 로직 30%+ 제거 실무 경험

문제

서비스마다 같은 변환 코드를 매번 복사·붙여넣기하고 있었습니다. 병원 시스템과 데이터를 주고받으려면 정해진 포맷(전문)으로 변환해야 하는데, 필드 하나 바뀌면 여러 파일을 동시에 수정해야 하는 구조였습니다. 전문 처리 자체를 제거할 수 없는 외부 의존성 제약이 존재했습니다.

해결

2~3명의 팀에서 프레임워크 설계와 구현을 주도했습니다. 전문 구조를 없앨 수는 없으니, 반복되는 변환 로직을 한 곳에서 관리하는 방향으로 잡았습니다.

프레임워크 내부에 MciUtils 클래스를 설계하여 input/output 매핑 로직을 공통 모듈로 추출하고, 각 서비스에서는 설정값만 넘기면 되도록 구조를 변경했습니다.

검증

프레임워크 업데이트 후 각 서비스 적용을 확인했습니다. 소스코드 라인 수 20~40% 절감, 동료들로부터 "코드 작성이 훨씬 수월해졌다"는 피드백을 받았습니다.

"제거할 수 없다면 다루기 쉽게 만들자."

전문 처리를 없애는 게 가장 깔끔하지만, 외부 API 의존성 때문에 불가능했습니다. 대신 팀원들이 변환 코드를 신경 쓰지 않아도 되는 구조를 만든 결과, 신규 서비스 추가 시 변환 로직 작성 시간이 체감상 절반 이하로 줄었습니다.

AI가 실무에 가까워지고 있는 지금, 코드는 사람뿐 아니라 AI가 읽고 이해하기 좋은 구조여야 한다고 생각합니다.

레거시 코드 SQL Injection 취약점 파악 및 개선 실무 경험

문제

SQLMap으로 테스트해보니 실제로 Injection이 성공하는 상태였습니다.

보안 관련 이슈를 접하고 궁금해서 SQL Injection을 학습한 직후, "우리 코드는 괜찮을까?"라는 생각에 레거시 코드를 분석했습니다.

사용자 입력값을 문자열 그대로 이어붙여 SQL 쿼리를 만드는 String Concatenation 방식이 여러 모듈에 걸쳐 수십 건 존재했고, ' OR '1'='1 같은 조건으로 인증 우회나 데이터 유출이 가능한 심각한 취약점이었습니다.

해결

2~3명의 팀 내에서 보안 점검과 개선을 자발적으로 주도했습니다. 점진적 전환과 전면 전환 중 고민했지만, 보안 취약점은 하나라도 남아있으면 의미가 없다고 판단하여 전면 개선을 택했습니다.

전체 코드베이스를 점검하여 PreparedStatement 기반으로 일괄 전환하고, 누락 방지를 위해 모듈별 체크리스트를 만들어 하나씩 소거했습니다.

검증

SQLMap으로 개선 전/후 비교 테스트를 수행했습니다. 개선 전 Injection 취약점이 감지되었으나, 개선 후 취약점 미탐지 및 Injection 시도 전면 차단을 확인했습니다.

보안 취약점은 하나라도 남아있으면 의미가 없다는 것을 배웠습니다.

점진적 전환과 전면 전환 중 고민했지만, 보안은 "거의 다 됐다"가 통하지 않는 영역이었습니다. 모듈별 체크리스트를 만들어 하나씩 소거하는 방식으로 누락을 방지한 경험이, 이후 다른 프로젝트에서도 전수 점검이 필요한 상황에서의 판단 기준이 되었습니다.

"배운 걸 써먹을 수 있을까?"가 아니라 "우리 코드는 괜찮을까?"로 연결되는 감각이 이때 생겼습니다.
